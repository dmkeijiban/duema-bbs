# 🔴 Codex 敵対的レビュー

「この設計は本当に大丈夫か？」を悪意ある第三者の視点で検証する。

## 使い方

```
/codex:adversarial-review pr=<番号>
/codex:adversarial-review file=<パス>
/codex:adversarial-review feature=<機能名>    # 機能名で広範囲をレビュー
```

## 実行手順

1. **対象を特定**
   - `pr=` → `gh pr diff <番号>` で差分取得
   - `file=` → ファイルを読む
   - `feature=` → 関連ファイルを `grep` で横断調査

2. **攻撃シナリオを列挙**（各項目について実際のコードを示して評価）

   **セキュリティ**
   - API エンドポイントへの認証なしアクセスは可能か？
   - SQL インジェクション・XSS・CSRF の余地はあるか？
   - 環境変数・API キーが露出するパスはないか？
   - RLS が無効化される迂回経路はないか？

   **データ整合性**
   - 並行リクエストでデータが壊れるケースはないか？
   - トランザクション不足で中途半端な状態になるケースはないか？
   - 外部キー・ユニーク制約・NULL チェックの抜けはないか？

   **パフォーマンス劣化**
   - N+1 クエリが発生するパスはないか？
   - インデックス未使用のフルスキャンはないか？
   - キャッシュ戦略がなく毎回重いクエリを叩くところはないか？

   **運用リスク**
   - デプロイ・ロールバック時にデータが壊れるケースはないか？
   - マイグレーション失敗時の復旧手順は存在するか？
   - エラーが無音で失敗する（例外が握り潰される）箇所はないか？

   **スケール限界**
   - データ量が10倍になったとき破綻する設計はないか？
   - 同時接続数が増えたときボトルネックになる箇所はないか？

3. **リスクを評価・分類**

   | リスクレベル | 基準 | 対応 |
   |---|---|---|
   | 🔴 Critical | 本番データ損失・認証バイパスの可能性 | 即対応必須 |
   | 🟡 High | パフォーマンス劣化・情報漏洩の可能性 | 今週中に対応 |
   | 🟢 Medium | 将来的な技術的負債になりうる | 次スプリントで対応 |
   | ⚪ Low | 気になるが許容範囲 | ドキュメントに残す |

4. **改善案を提示**
   - Critical/High のみ具体的なコード修正案を添付
   - Medium/Low は方向性のみ示す

## ルール

- 「問題なし」で終わらせない：必ず何らかのリスクを掘り出す
- コードを読まずに一般論を言わない：必ずコードの具体的な行を指摘する
- 修正コストと放置リスクのトレードオフを必ず示す
